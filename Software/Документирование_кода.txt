===== Докстринги =====

Сперва во всех файлах с кодом нужно прописать документацию в виде докстрингов — текстовые описания модулей, функций, классов и методов в тройных кавычках в одной строке или на нескольких. Они должны начинаться с условного «Этот метод…» и сразу рассказывать, что он делает, не вдаваясь в детальные детали:

{{{code: lang="python" linenumbers="True"
def register_user(name: str, email: str) -> bool:
	"""Регистрирует нового пользователя.

	Args:
		name: Имя пользователя.
		email: Электронная почта.

	Returns:
		True, если регистрация прошла успешно.
	"""
}}}

Без докстрингов экспортировать нечего, комментарии через ''#'' за документацию не считаются.

Формат докстрингов — markdown, поэтому можно вставлять заголовки, ссылки и изображения
* ссылка: ''[Zim](https://zim-wiki.org/)'' или просто https://github.com/zim-desktop-wiki/zim-desktop-wiki
* заголовок: ''##Текст''
* изображение: ''<img src="https://raw.githubusercontent.com/testitquickly/Software-Testing-Glossary/master/images/alexei_lupan.jpg" alt="Alexei Lupan" height="200" align="right" />''

===== Имя каталога проекта =====

Каталог с проектом в Python надо называть по шаблону ''word_word'' — это шаблон наименования пакетов. Детально об этих сущностях см. [[Python:Модули, пакеты и классы]]

Пакет — это условная сущность, в которой может быть (зачем-то) пусто, или там могут быть файлы (с кодом) и вложенные каталоги с файлами. Чтобы создать пакет, надо в каталоге с файлами создать пустой файл ''__init__.py''.

Условности:

* ''word_word'' — это имя пакета.
* файл ''__init__.py'' — модуль пакета ''word_word''.

С этого момента весь проект ''word_word'' можно будет импортировать в другой проект как обычный пакет, внутри которого есть функции, которые можно вызывать в своём коде…

И в корневом катлоге проекта тоже надо положить пустой файл ''__init__.py'' — теперь весь проект будет считаться пакетом Python.

===== Сборка документации =====

…ориентируется на пакеты и «модули в пакетах». 

Каталог считается пакетом, только если в нём есть файл ''__init__.py'', и только тогда будет прочитано содержимое модулей (''*.py''-файлов). Иначе все модули будут проигнорированы.

У сборщика есть параметр, который приказывает игорировать функции, классы и методы, у которых нет докстрингов (то есть без комментариев для документации).

===== Сборщики документации =====

Крайне разумно пользоваться автоматическими сборщиками документации. Даже если разработчик того или иного сборщика умрёт и проект зачахнет, всегда можно переключиться на аналогичный проект. 

==== pydoc ====

''pydoc -w main.py''

Самый простой инструмент, потому что умеет экспортировать в html только //один// файл за раз. В рекурсивность не умеет.

==== pdoc ====

Генератор документации по внутреннему программному интерфейсу (API) в связные html-страницы на основе информации из докстрингов по всем пакетам в проекте. 

Он //top// для контроля за содержанием и оформлением API проекта, но не более. Умеет только то, что было объявлено, добавить в его вывод API какие-то статьи из файлов md очень сложно (можно вставить как главную страницу README.md через его прописываение в __init__.py в корне проекта, но и только, потому что ''pdoc'' был сделан только для автоматической сборки информации из докстрингов. Настройки: минимальны. Обрабатывает все файлы проекта рекурсивно, полноценно поддерживает аннотацию типов, автоматически создает перекрёстные ссылки между элементами кода, включает встроенный веб-сервер с автообновлением и понимает докстринги в стилях numpydoc или Google.

Умеет рендерить диаграммы из [[GraphViz]] (но только без сложного оформления) и [[+Диаграммы Mermaid JS]]

См. [[+pdoc]]

==== MkDocs ====

Можно использовать рядом с pdoc.

//MkDocs// может собирать сайт из разрозненных статей, гайдов, туториалов из файлов Markdown, внутри которых можно автоматически подтягивать информацию из докстрингов (собрал всё на отдельной странице, топ). Богат на плагины и расширяемость функциональности. 

Впечатляет и радует, даже если требует какого-то времени на освоение и настройку.

Но есть серьезная засада. Проект развился так, что его стало невозможно развивать, поэтому его переписали чуть менее, чем полностью, и архитектура его v2 оказалась несовместимой с v1. Полностью отпадают все плагины, все настройки, происходит переезд с yaml на toml (что, конечно, хорошо), но в прод с ним идти невозможно, остается

* или использовать его «замороженную» версию MkDocs (например, 1.6) 
* или подождать, когда будет готов MkDocs v2
* или переехать на наследника — Zensical.

См. [[+MkDocs]]

==== Zensical ====

Наследник [[+MkDocs]], внешне почти идентичен, под капотом вообще другой зверь: 

* полноценный современный SSG (static site generator) с нуля, поддерживающий документацию и технические сайты.
* совместим с Material для MkDocs
* toml вместо yaml
* улучшенный клиентский поиск
* Open Source под лицензией MIT

Находится в разработке, поэтому ещё не является абсолютной заменой MkDocs.

См. [[+Zensical]]

==== Sphinx ====

Топ для крупных проектов, со сложной настройкой и множеством возможностей. Если взлетит, то будет очень гибко: можно комбинировать статьи, туториалы и API-документацию. Но ввиду сложной настройки в неумелых руках может больше раздражать, чем помогать.

Sphinx работает с текстами в формате reStructuredText, это «расширенный» Markdown, его надо изучить и понять отдельно. MD рядом с RST выглядит проще, но RST умеет больше, поэтому его сложность логична, он умеет делать сложную документацию с перекрёстными ссылками, автогенерацией индексов, сложные таблицы, PDF/ePub… Было придумано промежуточное решение, которое Sphinx понимает — MyST, a powerful authoring framework that supports blogs, books, articles and journals. На первый взгляд MyST ерундовая ерунда, но есть активное коммьюнити, так что не всё однозначно.

См. [[+Sphinx]]

==== Pandoc ====

Универсальный конвертер документов. Он может брать файлы в Markdown, RST, LaTeX, HTML, DOCX и др. и конвертировать в любой другой поддерживаемый формат — HTML, PDF, ePub, LaTeX и т.д. Он может собирать полноценную общирную документацию в виде (буквально) книг. Топ, если надо из проекта сделать, эээ, книгу, которую будут читать обыкновенные стейкхолдеры и обыкновенные заказчики. 

Обычно используется вместе с каким-то генератором файлов (pdoc, MkDocs или Sphinx), который всё собирает, после чего подключается Pandoc, который делает всю «книжную магию»: оглавления, шаблоны, типографику, LaTeX, pdf, ePub.

См. [[+Pandoc]]
