====== RSync ======

[ @rsync @backup @ssh ]

Remote Sync — инструмент для продвинутого копирования файлов. Первый раз копирует все файлы, при повторном запуске копирует только новые файлы и только те файлы, которые были изменены, что сильно снижает объемы копирования в будущем.

Можно синхронизировать содержимое каталогов на источнике и получателе, а можно просто докидывать файлы в удалённый каталог. Работает и по ssh. 

Нет версионности файлов, для этого см. BorgBackupб Restic, Rclone. Корректно говоря, RSync может делать инкрементальные бэкапы с дедупликацией данных с опцией ''--hard-links'' и ''--link-dest'', но там надо очень крепко подумать и понимать в [[Debian:HowTo:Сделать:Симлинки и хардлинки]]

===== Учесть =====

Когда надо скопировать сам каталог — указать каталог без закрывающего слэша.

Когда надо скопировать содержимое каталога — нужен закрывающий слэш.

Когда надо копировать миллионы файлов, RSync может задохнуться. Каждая проверка файла в RSync происходит через вызов stat() и выполнение сравнения метаданных, что вызывает «хаотичное» чтение диска в разных местах (болезненно для HDD). Миллион вызовов нагружает любой CPU и диски, особенно если они слабые (типично для NAS). Может быть разумно сперва собрать все файл в tar, передать как один файл, на получателе распаковать архив и разложить файлы по местам. Но так потеряется инкрементальность… Иногда проблема решается через много Gb RAM, иногда надо полностью перейти на SSD, иногда лучше перейти на другие файловые системы.

===== Параметры =====

* ''-a'' (или --archive): архивный режим, включает рекурсивную копию (-r) и сохранение прав доступа, владельцев, меток времени (-ptgoD)

* ''-u'' (или --update): Пропустить на получателе файлы, которые уже более свежие, чем пришло от отправителя. Очень помогает не затереть новые правки.

* ''-z'': сжимает данные при передаче, что экономит трафик и ускоряет копирование по сети, но очень вредит процессору при копировании уже сжатых файлов:
	* музыки (.mp3, .ogg, .flac…)
	* фото (.jpg, .png, .webp…)
	* видео (.mp4, .mkv…)
	* архивов (.zip, .tar.gz, .7z…)
	* и некоторые файлы pdf (тут уж непредсказуемо)
Это уже сжатые файлы, сжимать их снова нет смысла, а иногда это увеличивает их объем. Для их игнорирования использовать опцию ''--skip-compress''

* ''-v'' (--verbatim): показывать файлы, которые задействованы в копировании

* ''-P'' (--partial --progress): отображается прогресс копирования + возможность докачки, если произошел обрыв соединения, дико полезно при копировании крупных файлов. Но по сравнению с --verbatim выдаёт в консоль излишне много информации, поэтому не всегда нужен.

* ''-R'' (или --relative): нужен, если в один бэкап-каталог сохраняем сразу несколько каталогов из разных мест на диске. При одиночной команде копирования не нужен.

* ''--delete'': полная синхронизация каталогов в обе стороны, удаляет с NAS файлы, которые уже были в бэкапе, но были удалены на ноутбуке.

* ''--delete-delay'': чуть безопаснее — файлы удаляются во время их передачи, а не в её начале.

* ''--remove-source-files'': ОСТОРОЖНО! Переносит файлы с ноута на NAS (каталоги остаются).

* ''-e ssh'': заюзать ssh

* ''--skip-compress=$file_list'' — надо заранее сделать перечень типов файлов, которые не надо сжимать перед отправкой, потому что они уже представляют собой архивированный контент. Их дополнительное сжатие занимает ресурсы процессора, а на размер файлов это влияет незначительно. Например: 

''file_list="aac,apk,arj,avi,bmp,bz2,dmg,flac,flv,gif,gz,heic,ico,iso,jar,jpeg,jpg,m4a,mkv,mov,mp3,mp4,mpeg,mpg,mts,ogg,pdf,png,rar,swf,tar,webm,webp,wma,wmv,xz,zip,zst,7z"''

* ''--exclude'': проигнорировать файл или каталог. Примеры: 
	''--exclude='*.tc~''' 
	''--exclude='Notebooks/'''

Когда надо игнорировать файлы по расширению, надо прописывать их через отдельные параметры. Если их много, удобнее перечислять их не в одной строке, а через перенос:
	''--exclude='*.tc~'  \''
	''--exclude='*.pdf'''  

Между переносами строк через ''\'' нельзя дописывать комментарии — это будет воспринято как полноценное завершение строки. Следующая будет считаться как «возникла из ниоткуда» без связи с предыдущей.

==== Пароли ====

для подключения можно прописать в отдельном файле и подставлять их через ''--password-file''. Подключения по [[ssh]] надо делать через ключи доступа.

==== Симлинки ====

По-умолчанию опция ''-a'' подразумевает копирование симлинков as is. Если надо копировать файл, на который указывает симлинк, надо использовать опцию ''-L'' (она же ''--copy-links'').

===== Примеры =====

==== Копирование файлов ====

В RSync надо пристально следить за указанием слэшей после названия каждого каталога по-отдельности.

Предположим, что есть каталог ''a/файлы…,'' и надо скопировать все эти файлы из каталога ''$a'' в каталог ''$b'':

''rsync -auzv \''
	''--skip-compress=$file_list \''
	''"$a/" "$b"''

 На получателе будет так: 

''b/файлы_из_a…''

Если забыть про слэши:

''rsync -auzv \''
	''--skip-compress=$file_list \''
	''"$a" "$b"''

на получателе будет так: 

''b/a/файлы…''

Потому что каталог ''$a'' будет полностью скопирован в каталог ''$b''. 

К слову, форма 

	''"$a/" "$b"''

абсолютно идентична

	''"$a/" "$b/"''

и здраво привыкнуть к слэшам by default.

==== Копирование каталогов ====

Надо скопировать каталог ''a/123'' (со вероятными подкаталогами) в ''b/'':

''rsync -auzv \''
	''--skip-compress=$file_list \''
	''"$a/123" "$b"''

 На получателе будет так: 

''b/123/файлы_из_123…''



==== Перенос файлов/каталогов ====

Аксиома: RSync копирует только файлы. Если под источником есть подкаталоги (даже пустые), они тоже будут скопированы, потому что каталоги — это тоже файлы, просто особого типа. Из-за этой аксимоы при переносе файлов через RSync возникают нюансы. 

Перенос файлов выполняет параметр ''--remove-source-files'' — сперва файлы копируются на сторону получателя, затем уничтожаются на стороне источника:

''rsync -auzv \''
	''--skip-compress=$file_list \''
	''--remove-source-files \''
	''"$a/" "$b"''

Но копируются только файлы, и если в каталоге ''$a'' были подкаталоги, то RSync их оставит на месте. Файлы (рекурсивно) исчезнут, а все подкаталоги останутся.

Поэтому после переноса файлов надо отдельной командой удалить и подкаталоги:

''rsync -auzv \''
	''--skip-compress=$file_list \''
	''"$a/" "$b" \''
	''&& find "$a" -mindepth 1 -maxdepth 1 -exec rm -r {} +''

-mindepth 1 — пропустить сам каталог ''$a''.
-maxdepth 1 — проверить только первый уровень (т.е. непосредственно под ''$a'').

Если какие-то подкаталоги (''a/123'' и ''a/456'') надо оставить:

''find "$a" -mindepth 1 -maxdepth 1 ! -name 123 ! -name 456 -exec rm -r {} +''

''rsync -az -e ssh user@remote_host:/remote/path/ /local/path/''

==== Копирование “ноутбук ← NAS” ====

''rsync -az -e ssh user@remote_host:/remote/path/ /local/path/''

===== Слить несколько каталогов в один =====

Может понадобиться заливать в один и тот же каталог на NAS подкаталоги из нескольких разных мест. Для этого нужна последовательность команд rsync с копированием нескольких каталогов в один и тот же. Но если хотя в одной из этих команд будет опция жесткой синхронизации ''--delete'', то все предыдущие файлы в итоговом каталоге будут удалены. Каждая следующая команда будет затирать каталоги, которые принесла предыдущая команда.

Для этого есть обходной путь:

* сперва без ''--delete'' скопировать нужные каталоги в отдельный каталог в /tmp, 
* и уже оттуда синхронизировать всё с удаленным каталогом в NAS вместе с ''--delete''

Пример:

''mkdir -p /tmp/merged_notebooks/''
''rsync -a /home/hdd/Notebooks/ /tmp/merged_notebooks/''
''rsync -a /home/hdd/Dropbox/Notebooks/ /tmp/merged_notebooks/''
''rsync -az --delete -e "ssh -i $ssh_keys" /tmp/merged_notebooks/ $nas_books/Notebooks/''

Но это хорошее решение только для каталогов с //небольшим// размером. Для больших каталогов это будет напряжно. Лучше разделять всë в удаленном каталоге по подкаталогам, потом на стороне NAS физически переместить уже синхронизированные подкаталоги в общий корень, и уже потом синхронизировать каждый подкаталог отдельно с --delete.
