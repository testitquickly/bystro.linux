====== Виртуальные окружения ======

[ @python ]

Для управления виртуальным окружениями Python надо установить пакет ''venv'', а для шикарной жизни можно поставить также шрифты и документацию:

''sudo apt install python3-venv python3-full''

==== Установить пакеты Python в виртуальных окружениях ====

1. **Зайти** в каталог с будущим проектом (даже если в нём только один файл //test.py//)
2. **создать** виртуальное окружение со своим названием. Имя виртуального окружения — на своё усмотрение. Пусть будет ''venv_ProjectName'':

''python3 -m venv venv_ProjectName''

В каталоге проекта появится подкаталог с объявленным названием виртуального окружения, со служебными файлами внутри. 

Можно назвать виртуалку ''.venv_ProjectName'' — каталог будет «скрыт», как это делается в Linux.

В каталоге проекта Python можно создавать бесконечное количество виртуальных окружений (с разными именами).

3. **запустить** виртуальное окружение — буквально вызвать через bash исполнение файла activate:

''source venv_ProjectName/bin/activate''

В консоли имя окружения будет отображаться перед hostname до тех пор, пока оно не будет деактивировано.

4. **установить** нужные пакеты Python внутри виртуального окружения:

''python -m pip install feedparser''

Не надо использовать внутри виртуальных окружений вызов pip3 (вроде ''pip3 install feedparser''), потому что при возникновении некоторых неочевидных условий это может вызвать непредсказуемую установку пакетов Python в системные каталоги.

Все файлы пакетов будут установлены внутри активного виртуального окружения и доступны только внутри него. Конкретно — пакет ''feedparser'' установлен в каталог ''venv_RepublishMicroposts/lib/python3.13/site-packages''

Если эти же пакеты будут нужны в другом виртуальном окружении — их надо будет установить в каждом отдельном виртуальном окружении Python.

5. **выполнить** свои скрипты Python
6. **деактивировать** виртуальное окружение

''deactivate''

Это вызов метода из файла …''/bin/activate''

==== Из неочевидного ====

=== Нельзя переименовывать каталоги с виртуальными окружениями ===

Например, ''venvProjectName'' в ''venvProjectName123'', потому что при последующем вызове переименованного окружения всё равно будет инициализировано предыдущее окружение, а из-за конфликта имён весь софт, который уже был установлен внутри него через pip3, будет недоступен, и установка новых пакетов в поднятое виртуальное окружение будет невозможно. Если прям надо — следует удалить каталог с виртуальным окружением и создать новый. 

Что хорошо — если произошло случайное переименование каталога с виртуальным окружением, его можно переименовать обратно и всё заработает.

Также невозможно прямо перетащить уже установленные пакеты из одного виртуального окружения в другое. Виртуальное окружение привязано к пути Python интерпретатора внутри venv и к абсолютным путям в pyvenv.cfg и к установленным wheel’ам в site-packages.

Рабочий способ миграции:

1. зафиксировать зависимости из старого окружения

''source old_venv/bin/activate''
''pip freeze > requirements.txt''

2. создать новое окружение

''python3 -m venv new_venv''
''source new_venv/bin/activate''

3. установить всё заново в новом окружении

''pip install -r requirements.txt''

=== На сервере может не быть полноценного окружения ===

Делаешь

''python3 -m venv venv_Project''

а в ответ

Error: Command '['/home/USER/Project/venv_Project/bin/python3', '-Im', 'ensurepip', '--upgrade', '--default-pip']' returned non-zero exit status 1.

Решение — “user installation”, потому что “normal site-packages is not writeable”:

''python3 -m ensurepip''

''python3 -m venv --without-pip venv_Project''

''curl -sS https://bootstrap.pypa.io/pip/3.6/get-pip.py | python3''

''source venv_Project/bin/activate''

И уже внутри запущенной venv ((venv_Project) [user@hostname] ~/Project$) выполнить 

''pip install beautifulsoup4 requests''
