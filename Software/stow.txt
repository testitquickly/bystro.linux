====== stow ======

[ @console @stow ]

Это менеджер симлинков на конфигурационные файлы. Но не просто менеджер файлов, а инструмент, который зеркалит дерево каталогов пакета (файловую структуру) в целевую директорию (обычно $HOME) через симлинки на конфиг-файлы (см. [[Debian:HowTo:Сделать:Симлинки и хардлинки]]), начиная от корня target (обычно $HOME, но можно указать иное).

Конфигурационные файлы программ и скриптов находятся в разных каталогах, там и сям. Когда надо переустановить/перенести систему, бэкап всех конфигов и их последующий перенос в соответствующие каталоги быстро становится очевидной проблемой. Можно делать полный бэкап каталога профиля юзера, но не все конфиги находятся в профиле юзера. Можно сделать ловкий трюк: постоянно держать все конфиги всех программ в одном каталоге, а там, где они по-настоящему должны находиться, сделать симлинки на соответствующие конфигурационные файлы из нашего мега-каталога. У этого подхода есть минусы, но его плюсы перевешивают, потому что конфиги в едином каталоге можно экспортировать в репозитории, обновлять и раздавать на другие компьютеры.

Утилита stow расширяет идею симлинк-конфигов и делает это удобнее, чем может показаться. Надо не только перенести все конфиги в один каталог («источник истины» для stow; например “''~/.stow''”) , но и создать внутри него структуру расположения конфиг-файлов по каталогам относительно своего $HOME. 

Наглядно: https://youtu.be/mEUUNRQdJF8

Полный мануал: https://www.gnu.org/software/stow/manual/stow.html

===== Теория =====

Для правильного управления структурой каталогов stow использует понятие «__пакет__». Важно понять эту концепцию, потому что тут всё организовано через эти пакеты.

Пакетом называется каждый каталог “первого уровня” внутри источника истины (в моём случае “''~/.stow''”). Как человек назовет «пакет», так его stow и будет дальше использовать. Например:

''~/.stow/abc/.config/init.xyz''

Здесь “abc” — объявленный мною пакет. Позже stow будет зеркалить полную структуру этого пакета/каталога — будет сделан симлинк на полный путь по структуре каталогов, вплоть до файла с конфигом. В целом необязательно, но логично называть пакеты для stow по имени каталога или приложения, к которому они относятся.

В контексте одного пакета может быть перечислено несколько файлов из разных мест. Здесь stow “увидит” два пакета: ''abc'' и ''activitywatch'':

''~/.stow/abc/.config/init.xyz''
''~/.stow/abc/.local/share/kdenlive/profiles/customprofile0 (тут файл без расширения)''
''~/.stow/activitywatch/.config/activitywatch/aw-server/aw-server.toml''
''~/.stow/activitywatch/.config/activitywatch/aw-server/settings.json''
''~/.stow/activitywatch/.config/activitywatch/aw-watcher-afk/aw-watcher-afk.toml''

Нельзя положить в каталог stow какой-либо файл «без каталога/пакета». Например, конфиг профиля ''~/.bashrc'' нельзя просто объявить в ''~/.stow/.bashrc'', потому что файл на верхнем уровне не является пакетом, и stow будет нечего/некуда зеркалить. Сперва надо подложить его в условный пакет — название на своё усмотрение, например, “mybash”:

''~/.stow/mybash/.bashrc''

Здесь stow “увидит” пакет “mybash” и будет работать с содержимым этого пакета.

Тонкий момент — можно закатать в содержимом stow-пакета сразу целый каталог со всеми его подкаталогами и файлами, например весь ''~/.config'' со всеми его подкаталогами и файлами:

''~/.stow/myconfig/.config/''

Делать это надо обосновано. Тот же ''~/.config'' — это системный каталог, в файлы которого постоянно пишет почти весь софт, из-за непредсказуемости этого процесса (какой-то софт добавили, какой-то удалили) это может привести к непредсказуемому смешиванию конфигов разных программ или даже к их потере. А вот если есть свой набор конфигов и скриптов в условном каталоге ''myScripts'', в котором все файлы свои, нужные, подконтрольные, то пакет из файлов в определенном каталоге, и мы точно знаем, что делаем, тогда ок:

''~/.stow/myScripts/''

Симлинки будут созданы на каждый файл, который будет находиться внутри этого каталога и всех его подкаталогов. 

===== Занести файлы в stow =====

Точнее надо говорить «создать структуру файловой системы», а для этого надо скопировать или перенести конфиги в каталог ''~/.stow/''. Здесь
''-f'': принудительно
''-v'': verbose

''cp -f -v ~/my_app/.my_config/init.lua ~/.stow/my_app/.my_config/init.lua''
''mv -f -v /usr/share/alsa/alsa.conf.d/10-samplerate.conf ~/.stow/alsa/usr/share/alsa/alsa.conf.d/10-samplerate.conf''
''mv -f -v ~/myScripts ~/.stow/myScripts''

То есть, можно брать под контроль stow любые конфиги, которые могут понадобиться, в том числе и файлы, к которым имеет доступ только root — в норме это не требуется.

===== Управление симлинками =====

Указать stow, какие из этих файлов должны быть отзеркалены в виде симлинков. 

==== Массово реплицировать структуру пакетов ====

1) обязательно перейти в каталог .stow, потому что выполнение stow “через точку” подразумевает «из этого каталога»:

''cd ~/.stow''

2) выполнить массовое симлинкование файлов из всех пакетов:

''stow -v .''

==== Реплицировать отдельные пакеты ====

''stow -d ~/.stow -t $HOME my_app''
''stow -d ~/.stow -t $HOME activitywatch''
''sudo stow -d ~/.stow -t / alsa''

Здесь
''-d'': директория, где лежат пакеты (~/.stow), «источник истины»
''-t'': путь, от которого stow надо воссоздать структуру своих «пакетов». Обычно корень профиля
''название пакета stow'': который надо засимлинькать

В последнем примере обратить внимание на путь, от которого надо воссоздавать структуру пакета alsa. По недосмотру легко получить файл ''~/usr/share/alsa/alsa.conf.d/10-samplerate.conf'' — в своем профиле.

==== Убрать симлинки ====

''stow -D'' __''НазваниеПакета''__

==== Restow ====

Приказать stow заново перечитать структуру пакетов, если там были сделаны какие-либо изменения (удалить + заново применить):

''stow -R'' __''НазваниеПакета''__

===== Пакеты stow в git =====

Можно [[Debian:HowTo:Сделать:git:Удаленный репозиторий]] для содержимого каталога ''~/stow'', чтобы спокойно обмениваться конфигами между несколькими компьютерами.

Предположим, что есть несколько ноутбуков, и у них hostname вроде ''personalAcer'', ''jobAsus'' и так далее.

Надо будет сделать отдельные файлы для уникальных настроек для каждого отдельного компьютера:

''~/.stow/hostname_configs/personalAcer.config''
''~/.stow/hostname_configs/jobAsus.config'' и etc.

и вызывать содержимое каждого из них в первой строке файла ~/.bashrc через переменную:

''source $HOME/.stow/hostname_configs/${HOSTNAME}.config''
