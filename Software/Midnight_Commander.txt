====== Midnight Commander ======

[ @mc @shortcuts ]

===== Установить =====

Debian: он есть на установочном диске

''sudo apt install mc''

Manjaro:

''sudo pacman -S mc''

===== Основные шорткаты =====

Все они прописаны в файле ''/etc/mc/mc.keymap'' Учесть, что глобальные шорткаты KDE перекрывают внутренние.

* ''Alt+I'' открывает ту же директорию, в которой находимся, в соседней панели (только англ раскладка)
* ''Alt+O'' открывает директорию под курсором в соседней панели
* ''Alt+C'' смена рабочего каталога через всплывающее окно; недоступен автокомплит через [''Tab]''
* ''Ctrl+O'' сворачивает/разворачивает MC
* ''Ctrl+Space'' показывает условный размер каталога под курсором
* ''Ctrl+\'' открывает меню «Каталоги быстрого доступа»
* ''Esc, Shift+E'' — история файлов, которые были открыты на редактирование (перенести курсор на нужный и нажать F4)
* ''mc -F'' показывает каталоги, в которых хранятся настройки

Для некоторых команд указан шорткат через букву ''M''. Например, для поиска файлов “F9 > Команда > Поиск файла” указан шорткат “''M-?''”. Неочевидно, что вместо буквы ''M'' надо нажать клавишу ''[Esc]'' и отпустить, затем нажать клавишу ''[?]'' — откроется окно настроек поиска файлов.

Для команд с шорткатом через букву ''C'' надо нажимать клавишу ''Ctrl''.

Единственное, чего не хватает — переименования файла под курсором по ''[F2]''. Надо вызывать окно перемещения файла через ''Shift+F6'', чтобы получить название файла, и отредактировать его по своему усмотрению.

===== Поменять систему отображения размера файлов =====

''F9''
''> выбрать левую или правую панель''
''> открыть Listing format... (Формат списка)''
''> выбрать User defined (Определённый пользователем)''
''> в предложенной строке уточнить размер — “size:4”''

Удобнее всего:

''type | name | size:4 | perm:4''

В первой колонке будет указан тип файла/каталога. Если это [[Debian:HowTo:Сделать:Симлинки и хардлинки]] — в первой колонке будет отображаться соответствующий знак.

В правой колонке будут показаны права файла в виде „rwx”.

Размер каждого файла будет округлен вверх. Поначалу выглядит непривычно и очень мешает точному подсчету размера файлов. Но если выставить “''size:4''” или “''size:7''” (чем больше число, тем точнее отображаемый размер), то мы будем уходить в сторону битов, что тоже неудобно для восприятия.

Для удобства можно в том же окне формата списка в поле «Строка мини-статуса в формате пользователя» (она же «Пользовательская строка состояния») установить параметр “''size:6''” или “''size:8''”, а также вывод прав в цифре:

''name | size:6 | mode:3''

Сохранить настройку:

''F9''
''> Сохранить настройки''

Также можно это всё настроить через

''mcedit ~/.config/mc/panels.ini''

Найти две строки с ''user_format= entries'' — одну под [New Left Panel] и вторую под [New Right Panel]. Прописать им

''user_format= name | size:4 | perm:4''

Для ''sudo mc'' все эти настройки надо повторить.

===== User Menu =====

User Menu вызывается по [F2]. Его содержимое можно переписать «под себя».

Например, надо часто открывать новое окно Konsole с уже открытым MC внутри. Это можно делать одной командой

''nohup konsole --hold -e mc >/dev/null 2>&1 &''

Можно
1. свернуть MC (через ''Ctrl+O'')
2. выполнить команду (или написать её полностью, или сделать для неё алиас в файле ''~/.bashrc'' и вызывать её командой-алиасом) и получить новое окно Konsole с открытым MC и сделать всё, что нужно
3. вернуться в первое окно
4. развернуть MC (''Ctrl+O'')

Алиасы из ''~/.bashrc'' внутри MC недоступны, потому что они являются частью окружения bash. При запуске MC в консоли мы «выходим» из окружения bash, MC запускается вместо текущей оболочки (бывает не только bash, есть и другие) и «занимает» действующий терминал, полностью перехватывая ввод/вывод. Поэтому внутри Konsole и внутри MC работают разные окружения, которые пересекаются только условно, окольными путями.

В MC можно 
1. вызвать командную строку через ''Alt+Shift+!''
2. написать в ней нужную команду. Или написать полный путь к отдельному исполняемому sh-файлу, в который записан нужный скрипт

Проще записать нужную команду в User Menu в MC. Если нужна не отдельная команда, а чуть более сложная цепочка команд, то можно закатать всё в исполняемый sh-файл и прописать в User Menu обращение к нему.

==== Настроить распознавание файлов по расширению ====

''F9'' 
''> Команда'' 
''> Править файл расширений''

Откроется файл ''$HOME/.config/mc/mc.ext.ini''

Настройки для tex и markdown в файле уже есть.

Файлы можно сопоставлять через расширение или через регэксп. 

“Include=editor” принудительно тянет nano/mcedit, можно убрать, эта настройка обычно перетирает заданное в Open.

ShellIgnoreCase=true означает, что расширение файла сравнивается без учёта регистра. Полезное.

''[sh]''
''Shell=.sh''
''ShellIgnoreCase=true''
''Open=kate %f &''

''[txt]''
''Shell=.txt''
''ShellIgnoreCase=true''
''Open=kate %f &''

''[md]''
''Shell=.md''
''ShellIgnoreCase=true''
''Open=kate %f &''

''[tex]''
''Shell=.tex''
''Open=kate %f &''
''ShellIgnoreCase=true''

''[markdown]''
''Regex=\\.mk?d$''
''Open=kate %f &''
''RegexIgnoreCase=true''

==== Отредактировать файл User Menu ====

''F9'' 
''> Команда'' 
''> Редактировать файл меню''
''> [Пользовательский]''

Если открыть «Местный» — будет сгенерирован новый файл с дефолтным содержимым. Это не наш метод.

Или отредактировать его напрямую:

''mcedit ~/.config/mc/menu''

Вставить в него любые свои команды по данному шаблону:

''h открыть копию MC в новой вкладке Konsole''
''nohup konsole --hold -e mc >/dev/null 2>&1 &''

	''# Create a new HTML file''
''H New HTML file''
	''{ echo "<html>"''
	''echo "\t<head>\n\t</head>"''
	''echo "\t<body>\n\t</body>"''
	''echo "</html>"; }  > new_page.html''

''NAS video''
	''/home/myScripts/nas/nas-mount-v1.sh''
	''/home/myScripts/nas/nas-mount-v2.sh''
	''/home/myScripts/nas/nas-mount-v3.sh''

Первая буква в строке перед названием пункта меню — шорткат для выполнения команды. Необязательна. Но если будет, то учесть, что одна и та же буква в разных регистрах — разные шорткаты.

Последовательность расположения команд — произвольная, в итоговом списке по [F2] они будут отображаться буквально одна за другой. 

Комментарии закрыты за символом ''#''

Максимальная длина названия команды: 60 символов.

Примеры оформления команд: [[+MC и user menu]]

===== mcedit =====

Есть смысл объявить mcedit основным редактором в терминале своего профиля:

''echo "export EDITOR=mcedit" >> ~/.profile && echo "export EDITOR=mcedit" >> ~/.bashrc && source ~/.profile && source ~/.bashrc''

Настроить mcedit — открыть любой файл на редактирование (''[F4]''):

''> F9'' 
''> Настройки'' 
''> Общие''

* Режим переноса строк: Нет (хотя в некоторых случаях он прям будет нужным)
* Прочие настройки:
	* Автоотступ "Вводом": Да
	* Подтверждать запись файла: Да
	* Сохранять позицию в файле: Да
	* Подсветка синтаксиса: Да
	* Постоянные блоки: Да
	* Групповая отмена: Да

«Постоянные блоки» — лучше оставить включенным. 

В редакторе mcedit выделенный текст постоянно остаётся выделенным, пока выделение не будет осознанно снято юзером (в любом месте текста нажать ''Shift+Rigth+Left''. Это привет из времен, когда компьютеры уже были, а мышек ещё не было — данный режим редактирования позволяет выделить буквы/строки в одном месте файла текст, перейти в другое место файла и скопировать (F5) или переместить (F6) выделенный текст под курсор, или по F8 удалить выделенные символы/строки. 

Но по-умолчанию выделение строк сохраняется, потому что а вдруг тебе захочется те же строки многократно копировать в разные места файла? Если не привыкнуть к этой особенности mcedit, местами будет конфуз из-за лишнего трогания клавиши F8. Ничего же не подсказывало, что выделение со строк не снято, а строки уже где-то далеко внизу экрана.

Ещё в mcedit не работает уже привычный Ctrl+/Ctrl+V, потому что это наследие более позднего Windows — MC не сохраняет информацию в стандартный xclip/xsel буфер системы.

Ещё надо знать, что максимальный размер редактируемого файла: 64 Мб. Если больше, вероятны потери данных.

=== Изменить содержимое “Author description header” ===

''F9''
''> Настройки''
''> Файл меню''
''> [Пользовательский]''

* [Местный] — в текущем каталоге будет создан файл “''.cedit.menu''”. Разумно для разных проектов.
* [Пользовательский] — редактируем файл “''$HOME/.config/mc/mcedit/menu''”

В этом файле шаблоны, которые автоматически применяются в зависимости от расширения файла, который открывается на редактирование. В MC menu есть шаблон “''unknown template''”, но он применяется только к файлам без расширения. 

Поэтому для редактирования файлов .sh применяется раздел 
''# ----- Begin [shell] language template -----''

В его секции Description можно отредактировать шаблон данных. Там надо сразу заменить строку 
''AUTHOR="`awk -F: '$3 == '$MY_UID' {print $5}' /etc/passwd`"''
на
''AUTHOR=$(awk -F: -v uid="$MY_UID" '$3==uid {split($5,a,","); print a[1]}' /etc/passwd)''

Информацию об авторе файла будет взята из ''/etc/passwd'', а там древний шаблон информации о юзере: //Full Name,Room,WorkPhone,HomePhone,Other//. В наше время при создании юзеров всё это не используется (см. [[Debian:HowTo:Сделать:User:Добавить]]), а пустующие места в шаблоне будут показаны так:

''Alexei Lupan, , ,'' 

После замены шаблона из ''/etc/passwd'' будут прочитываться данные до первой запятой, а на нас снизойдет админская благодать или озарение о том, что имя юзера можно написать самому, а не дергать каждый раз ''/etc/passwd''.

Полезно добавить в шаблон свои емайл, телефон и домашний адрес, потому что код надо писать так, словно его будет ревьюить маньяк, который знает, кто ты и где ты живёшь.

=== Сделать шаблоны для других расширений файлов в файле “menu” ===

Для всех нужных расширений файлов надо делать отдельные записи. Все они должны быть вставлены ДО начала ''#---- Begin common section -----''

Очень важно: не вставлять пустую строку после ''cat >>%b <<EOF'' — вместо шаблона будет приходить пустота.

Шаблон короткий:

''#------------------------- Begin txt template -----------------------------''
''+ y Text\ File | f \.txt$''
''1       Description header''
		''unset LANG''
		''unset LANGUAGE''
		''LC_ALL=''
		''MY_UID="`id | sed 's/^.*uid=\([^(]*\).*$/\1/'`"''
		''cat >>%b <<EOF''
		''# Alexei Lupan, `date +%%Y`''
		''# Email: myemail@testitquickly.com''
		''# Hostname: `uname -n`''
		''#''
		''# Этот файл''
		'': << 'COMMENT'''
		''…''
		''COMMENT''

		''#----------------------------------------------------------------------''
		''#''
		''# main()''

		''EOF''
''#--------------------------- End txt template -----------------------------''

Шаблон для файлов без расширения называется 

''# ----- Begin unknown template -----''
 
Его лучше убрать, потому что с ним в файлах с нестандартным расширением (*.fn) при вызове меню пользователя будут два вхождения для Description header — для файлов fn (я добавил) и для unknown files. А если его не будет, и я открою файл с нестандартным расширением, и захочу туда вставить Description header, для которого еще не добавлен отдельный шаблон, то в меню просто не будет пункта Description header, и это будет подсказкой «Сбегай добавь».

==== Подсветка текста ====

По-умолчанию при редактировании файлов MC определяет их тип и включает ту или иную подсветку текста. Я держу некоторые настройки или скрипты для Bash в файлах с расширением ''var'' или fn, и хочу при их редактировании видеть привычную подсветку для скриптов Bash.

По-отдельности в каждом открытом на редактирование файле можно выбрать стиль подсветки через ''F9 > Настройки > Подсветка синтаксиса'', но эта настройка слетает сразу после закрытия файла.

Можно сделать свой файл подсветки для других расширений, но есть подход проще: включать подсветку для скриптов sh во всех файлах с «неизвестным» расширением.

Скопировать общий файл подсветки в свой профиль (по-умолчанию он там отсутствует):

''cp /usr/share/mc/syntax/Syntax $HOME/.local/share/mc/syntax/Syntax''

Открыть файл, спуститься к последним строкам и заменить

''file .\* unknown''
''include unknown.syntax''

на

''file .\* unknown''
''include sh.syntax''

Сохранить, изменения подхватятся моментально.

Теперь редактирование файла синтаксиса будет доступно через ''F9 > Настройки > Редактирование файла синтаксиса''

Сделать то же самое для файлов с «уже известным расширением» вроде .txt (по-умолчанию такие файлы не перечислены в перечне файлов синтаксиса):

''file ..\*\\.txt$ txt\sFile''
''include sh.syntax''

==== Граница переноса строк ====

В норме перенос на 72 символе, но если надо переносить, например, на 96 символов:

''mcedit ~/.config/mc/ini''

''editor_word_wrap_line_length=96''

Принудительный перенос слов в строке (wrap text) делается через 
''F9'' 
''> Форматирование''
''> Форматировать абзац (Alt+p)''

==== Сортировка строк ====

Выделить нужные строки.

''F9'' 
''> Форматирование''
''> Сортировать (Alt+t)''

Появится строка для ввода параметров сортировки. Мы подразумеваем //естественную// сортировку: ''-V'' (оно же --version-sort), natural sort of numbers within text. Это когда:

Числа внутри строк трактуются как числа, а не текст:

* a1
* a2
* a10

Заглавные буквы идут раньше строчных:

* A
* B
* a

Латиница перед кириллицей:

* a
* b
* а
* б

Корректно-машинной считается //алфавитная// сортировка, но она всегда раздражает.

Если в выборке для сортировки есть пустые строки — они будут однозначно удалены.

Хорошо бы посмотреть весь ''man sort'' в отдельном окне. 

==== Список экранов ====

В общем меню есть Команда > Список экранов с подсказкой о том, что шорткат “''M-`''”. 

Открыть в MC какой-нибудь файл на просмотр (''[F3]'').

Нажать [''Esc]'' затем [`]. Откроется список действующих экранов:
* Панели (каталог, в котором мы находились)
* Просмотр (путь к файлу)

Можно перейти к «Панели» — откроется экран MC. Можно перейти в другой каталог к другому файлу и открыть его, например, на редактирование (''[F4]'').

Нажать [''Esc]'' затем [`]. Откроется список действующих экранов, которых теперь три:

* Просмотр (путь к первому файлу)

==== Копировать текст из одного файла в другой ====

Шорткаты не работают. И копировать можно текст только из файла, открытого в режиме редактирования. Исключение — если эмулятор терминала позволяет выделять текст мышкой, но это уже подключение современности.

Кажется, что остается только дописывать содержимое одного файла в другой, вручную удалять ненужное и оставлять нужное. Но ведь файлы могут быть неограниченного диаметра… Решение неочевидное, но простое.

1. Открыть в MC какой-нибудь файл на редактирование (''[F4]'').

2. Нажать [''Esc]'' затем [`]. Откроется список действующих экранов:

* Панели (каталог, в котором мы находились)
* Просмотр (путь к файлу)

3. Перейти к «Панели» — откроется экран MC. 

4. Перейти в другой каталог к другому файлу и открыть его на редактирование (''[F4]'').

5. Нажать [''Esc]'' затем [`]. Откроется список действующих экранов, которых теперь три:

* Панели (каталог, в котором мы находились)
* Просмотр (путь к первому файлу)
* Правка (путь ко второму файлу)

6. Перейти к файлу, из которого надо сохранить текст. Выделить нужный абзац:

''> F9'' 
''> Правка''
''> Копировать в файл буфера обмена''

Файл буфера обмена — это буквально отдельный файл, который находится в памяти, хотя должен был бы находиться в /tmp, он не сохраняется в стандартный xclip/xsel буфер системы, поэтому нельзя обычным Ctrl+V вставить куда-то его содержимое.

7. Нажать [''Esc]'' затем [`], перейти ко второму файлу:

''> F9'' 
''> Правка''
''> Вставить из файла буфера обмена''

Сохраниться.

	Учесть, что .bashrc should be used only for Bash-specific settings (e.g., shell options, aliases, etc.) while .profile should be used for setting environment variables like EDITOR (which is used by programs such as sudoedit, git, etc.).
	
	a shell is a program designed to start other programs.
	
	Bash is a Bourne-like shell. It reads commands from ~/.bash_profile when it is invoked as the login shell, and if that file doesn't exist¹, it tries reading ~/.profile instead.
	
	Therefore:
	
	~/.profile is the place to put stuff that applies to your whole session, such as programs that you want to start when you log in (but not graphical programs, they go into a different file), and environment variable definitions.
	
	~/.bashrc is the place to put stuff that applies only to bash itself, such as alias and function definitions, shell options, and prompt settings. (You could also put key bindings there, but for bash they normally go into ~/.inputrc.)
	
	~/.bash_profile can be used instead of ~/.profile, but it is read by bash only, not by any other shell. (This is mostly a concern if you want your initialization files to work on multiple machines and your login shell isn't bash on all of them.) 	On modern unices, there's an added complication related to ~/.profile. If you log in in a graphical environment (that is, if the program where you type your password is running in graphics mode), you don't automatically get a login shell that reads ~/.profile. Depending on the graphical login program, on the window manager or desktop environment you run afterwards, and on how your distribution configured these programs, your ~/.profile may or may not be read. If it's not, there's usually another place where you can define environment variables and programs to launch when you log in, but there is unfortunately no standard location.
	
	Note that you may see here and there recommendations to either put environment variable definitions in ~/.bashrc or always launch login shells in terminals. Both are bad ideas. The most common problem with either of these ideas is that your environment variables will only be set in programs launched via the terminal, not in programs started directly with an icon or menu or keyboard shortcut.
