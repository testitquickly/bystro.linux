====== Удаленный репозиторий ======

[ @git ]

Предположим, есть доступ к удаленному серверу по [[Software:ssh]] без ''sudo'', и надо там сделать свой репозиторий, чтобы синхронизировать между ноутбуками конфигурационные файлы, которые собраны под руководством [[Software:stow]]

===== Безопасность =====

Сперва надо осознать, что взломать можно всё, особенно если оно выставлено в сеть или находится на неизвестном компьютере (у хостера) и принять эту простую истину.

Шифровать файлы в репозитории через [[Software:GPG]] — можно, но не нужно, это сразу отметает возможность делать diff. GPG в репозиториях используется для удостоверения подписей коммитов, не более. Через ''git config --global commit.gpgsign true'' можно включить проверку каждого автора коммита и защиту от подмены истории. Но это не шифрует данные.

Закатывать файлы в запароленный архив — но тогда зачем git? Достаточно Dropbox и возни с архивом на каждом ноуте.

Держать репозиторий в зашифрованной партиции на сервере — можно, но только если значительное повышение нагрузки на CPU сервера действительно необходимо и хостер это одобрил.

Защитить доступ к репозиторию от взлома методом перебора паролей — вот это можно. На удаленном сервере для каждого юзера, которому будет разрешен доступ в будущий репозиторий, надо делать отдельный ключ ssh для каждого (проще шарить в команде один ключ, но лучше делать отдельные и настаивать на том, чтобы каждый использовал собственный ключ доступа). Это не охранит нас от тех, кто может по неведению/глупости передать свой ключ ssh злым юзерам, но шоподелать. Также подумать про неизбежность невнимательного коммита всех этих наших ключей доступа. Тщательно следить за тем, чтобы файлы каталог с ключами (и ssh, и api, и от квартиры) был прописан в ''.gitignore''

==== Ограничить доступ по паролю к серверу ====

В файле ''/etc/ssh/sshd_config'' отключить доступ по паролю (останется только доступ по ключу):

''PubkeyAuthentication yes''
''PasswordAuthentication no''
''PermitRootLogin no''

==== Сделать новый ключ SSH для репозитория ====

На ноуте:

''ssh-keygen -t ed25519 -f'' __''каталогХраненияКлючей/названиеКлюча''__

Получить в итоге два файла:

private key → хранится только у себя клиента
public key → лежит на сервере в ''~/.ssh'' и упомянут в списке ''authorized_keys''. 

Назовем его “key-for-git”.

==== Ограничение SSH-ключа в репозитории ====

Эта инструкция запрещает shell-доступ на сервер с ключом "для репозитория", с ним будут доступны только git-операции.

''mcedit ~/.ssh/authorized_keys''

В файле уже (вероятно) должен быть добавлен общий ключ пользователя на сервере.

Вставить новую строку:

''no-port-forwarding,no-agent-forwarding,no-X11-forwarding'' __''публичныйКлюч_key-for-git''__

Здесь:

**command** — принудительно запускает git-shell, а не обычный shell
**no-port-forwarding** — запрещён проброс портов
**no-agent-forwarding** — запрещён SSH-agent forwarding
**no-X11-forwarding** — запрещён X11 forwarding
**публичныйКлюч** — полное содержимое файла с публичным ключом ssh (“key-for-git”); начинается с ssh-ed25519 (тип ключа; алгоритм Ed25519) и длинная последовательность символов...

Сохранить, выйти.

Из нового окна Konsole проверить доступ с этим ключом: 

''ssh -i $MY_KEYS/key-for-git -p 12345 username@host''

В норме — сразу произойдет логин. Дальше молиться кому угодно о том, чтобы ограничения сработали.

===== Сделать удаленный репозиторий =====

Залогиниться на удаленный сервер через SSH, перейти в каталог, в котором будет постоянно находиться репозиторий. Его можно сделать где угодно в профиле (игнорируем //public_html//).

Например'':''

''mkdir -p ~/git/myConfigs && cd ~/git/myConfigs && git init''

==== Изменить права на каталог репозитория ====

Из-под своего юзера на сервере:

''chmod -R go-rwx ~/git''

chmod — изменяет права доступа к файлам и каталогам.
-R — рекурсивно, то есть для всех файлов и папок внутри ~/git.
go-rwx — убрать права у групп (g) и остальных пользователей (o):
r — чтение
w — запись
x — выполнение

То же самое:

''chmod 700 ~/git''

Итог: у группы и всех остальных пользователей отнимаются все права доступа к содержимому ~/git. Владелец (user, u) не затрагивается.

==== Сделать ветки и начальный коммит ====

Перейти в каталог репозитория:

''cd ~/git/myConfigs''

Технически создать репозиторий git:

''git add .''

Сконфигурировать основного пользователя репозитория:

''git config --global user.email "astenix@testitquickly.com" && git config --global user.name "Alexei Lupan"''

Сделать начальный коммит в репозиторий (технический шаг, файлы не нужны):

''git commit -m "init"''

Переключиться на ветку “master”:

''git branch -M master''

Если нужно будет делиться этим репозиторием с кем-то — сделать его описание:

''touch README''

==== На ноуте ====

''cd ~/myConfigs''

Клонировать репозитории надо в пустой каталог, поэтому весь каталог, в котором будут постоянно находиться файлы репозитория (''~/myConfigs'') надо временно перенести в другое место.

Дальше надо точно знать параметры подключения к своему ''user@server'' — адрес, порт, логин. Мне для подключения надо указать:

* место хранения ssh ключа для git
* username
* host с портом
* полный путь к месторасположению каталога с репозиторием на сервере
* название каталога на ноуте, в котором будет лежать репозиторий (''~/myConfigs''):

''GIT_SSH_COMMAND="ssh -i $MY_KEYS/key-for-git" git clone ssh://username@host:12345/home/myUsername/git/myConfigs ~/myConfigs''

Появляется каталог ''~/myConfigs'' и в нем дежурный подкаталог ''.git''.

Добавить файл

''mcedit ~/myConfigs/.gitignore''

и записать в него минимальный набор каталогов и файлов, которые надо игнорировать.

''# каталоги''
''abc/''

''# временные файлы''
''*~''
''*.bak''
''*.tmp''

''# системные файлы''
''Thumbs.db''

''# логи''
''*.log''

Commit:

''git add .gitignore''
''git commit -m "added gitignore"''
''git push origin master''
''git status & git log''

Если всё ок — вернуть в ''~/myConfigs'' все файлы, которые надо держать в удаленном репозитории. Затем:

''git add --all''
''git commit -m "все файлы"''
''git push origin master''
''git status & git log''

==== Мягкое ограничение некоторых файлов ====

Вероятно, какие-то конфигурационные файлы надо держать в репозитории, но не надо регистрировать каждое их обновление. Например:

''myStow/uss-Asus-Zenbook-17/.local/share/mc/filepos''
''myStow/uss-Asus-Zenbook-17/.local/share/mc/history''

Полный путь к файлам в репозитории можно проверить через 

''git ls-files''

В git их можно не игнорировать (жестоко выкинуть из индекса), а всего лишь слегка «придушить»:

''git update-index --skip-worktree'' __''путь от корня репозитория к файлу''__

Пример:

перейти в каталог с репозиторием и выполнить

''git update-index --skip-worktree myStow/uss-Asus-Zenbook-17/.local/share/mc/filepos''
''git update-index --skip-worktree myStow/uss-Asus-Zenbook-17/.local/share/mc/history''
''git update-index --skip-worktree myStow/uss-Asus-Zenbook-17/.config/mc/panels.ini''
