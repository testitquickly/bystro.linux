====== Удаленный репозиторий ======

[ @git ]

Предположим, есть доступ к удаленному серверу по [[Software:ssh]] без ''sudo'', и надо там сделать свой репозиторий, чтобы синхронизировать между ноутбуками конфигурационные файлы, которые собраны под руководством [[Software:stow]]

===== Безопасность =====

Сперва надо осознать, что взломать можно всё, особенно если оно выставлено в сеть или находится на неизвестном компьютере (у хостера) и принять эту простую истину.

Шифровать файлы в репозитории через [[Software:GPG]] — можно, но не нужно, это сразу отметает возможность делать diff. GPG в репозиториях используется для удостоверения подписей коммитов, не более. Через ''git config --global commit.gpgsign true'' можно включить проверку каждого автора коммита и защиту от подмены истории. Но это не шифрует данные.

Закатывать файлы в запароленный архив — но тогда зачем git? Достаточно Dropbox и возни с архивом на каждом ноуте.

Держать репозиторий в зашифрованной партиции на сервере — можно, но только если значительное повышение нагрузки на CPU сервера действительно необходимо и хостер это одобрил.

Защитить доступ к репозиторию от взлома методом перебора паролей — вот это можно. На удаленном сервере для каждого юзера, которому будет разрешен доступ в будущий репозиторий, надо делать отдельный ключ ssh для каждого (проще шарить в команде один ключ, но лучше делать отдельные и настаивать на том, чтобы каждый использовал собственный ключ доступа). Это не охранит нас от тех, кто может по неведению/глупости передать свой ключ ssh злым юзерам, но шоподелать. Также подумать про неизбежность невнимательного коммита всех этих наших ключей доступа. Тщательно следить за тем, чтобы файлы каталог с ключами (и ssh, и api, и от квартиры) был прописан в ''.gitignore''

==== Сделать новый ключ SSH ====

На ноуте:

''ssh-keygen -t ed25519 -f ~/.ssh/id_git''

Получить в итоге два файла:

private key → хранится только у клиента (например в файле ''~/.ssh/id_git'', но можно положить его куда угодно)
public key → лежит на сервере в authorized_keys

===== Сделать удаленный репозиторий =====

Залогиниться на удаленный сервер через SSH, перейти в каталог, в котором будет постоянно находиться репозиторий. Его можно сделать где угодно, ему необязательно находиться в //public_html//.

==== Ограничить доступ по паролю к серверу ====

В файле /etc/ssh/sshd_config отключить доступ по паролю (останется только доступ по ключу):

''PubkeyAuthentication yes''
''PasswordAuthentication no''
''PermitRootLogin no''

==== Создать репозиторий ====

''mkdir -p ~/git/mySettings.git && cd ~/git/mySettings.git && git init --bare''

==== Изменить права на каталог репозитория ====

Из-под своего юзера:

''chmod -R go-rwx ~/git''

chmod — изменяет права доступа к файлам и каталогам.
-R — рекурсивно, то есть для всех файлов и папок внутри ~/git.
go-rwx — убрать права у групп (g) и остальных пользователей (o):
r — чтение
w — запись
x — выполнение

Итог: у группы и всех остальных пользователей отнимаются все права доступа к содержимому ~/git. Владелец (user, u) не затрагивается.

==== Ограничение SSH-ключа в репозитории ====

Эта инструкция запрещает shell-доступ на сервер с ключом "для репозитория", с ним будут доступны только git-операции.

''mcedit ~git/.ssh/authorized_keys''

Вставить это:

''command="git-shell -c \"$SSH_ORIGINAL_COMMAND\"",no-port-forwarding,no-agent-forwarding,no-X11-forwarding ssh-ed25519 ABC...''

Здесь:

command="git-shell -c \"$SSH_ORIGINAL_COMMAND\"" — принудительно запускает git-shell, а не обычный shell
no-port-forwarding — запрещён проброс портов
no-agent-forwarding — запрещён SSH-agent forwarding
no-X11-forwarding — запрещён X11 forwarding
ssh-ed25519 — тип ключа (алгоритм Ed25519)
ABC... — сам публичный ключ, закодированный в Base64

==== На ноуте ====

cd ~/.mySettings
git init
git add .
git commit -m "init"
git remote add origin user@server:~/git/mySettings.git
git push -u origin main

==== Настройка клиента (~/.ssh/config) ====

Можно обще, через файл 

''Host mygit''
	''HostName server_ip''
	''User youruser''
	''IdentityFile ~/.ssh/id_git''

4) Использование

''git remote add origin mygit:~/git/mysettings.git''

Итог:

обычный SSH остаётся полным (для администрирования)
git-ключ ограничен только git-shell
компрометация git-ключа не даёт shell-доступ

А можно очень точно:

настраивать ~/.ssh/config не обязательно.

Можно указывать ключ прямо в команде.

Примеры:

git clone

GIT_SSH_COMMAND="ssh -i ~/.ssh/id_git" git clone user@server:~/git/mysettings.git

git pull / push

GIT_SSH_COMMAND="ssh -i ~/.ssh/id_git" git pull
GIT_SSH_COMMAND="ssh -i ~/.ssh/id_git" git push

Технически:
всё работает без конфигурации клиента
ключ можно не прописывать глобально

Если с репозиторием взаимодействовать через какой-то софт ([[Software:SmartGit]] или [[Software:VSCodium]]), надо подумать про настройку его внутреннего механизма работы с SSH — встроенный SSH + ключ.

В SmartGit:

Preferences → SSH Client
Выбрать:
Built-in SSH или System SSH (лучше System SSH)
Указать ключ через env-переменную или конфиг

Что-то вроде
Preferences → Git → SSH
и указать точное расположение нужного ключа:

''ssh -i ~/.ssh/id_git''

Или можно

~/.ssh/config под репозиторий

Вставить

''Host mygit''
	''HostName server_ip''
	''User youruser''
	''IdentityFile ~/.ssh/id_git''

В SmartGit:

''mygit:~/git/mysettings.git''

Ключ подхватывается автоматически.

==== Собрать в отдельный каталог структуру файлов для stow ====

Внутри stow-каталога ты создаёшь структуру, которая имитирует корень системы:

'''
mySettings/
  home/
	astenix/
	  newFolder/
		File.fn
'''

Затем выполняешь:

'''
stow -t /home/astenix mySettings
'''

stow проходит по дереву mysettings
для каждого файла создаёт симлинк в -t (target)
утилита создаёт промежуточные директории, если их нет
и даже не требует, чтобы файл уже существовал в системе. Он просто появится как симлинк вроде такого::

''/home/astenix/newFolder/File.fn -> ~/mysettings/home/astenix/newFolder/File.fn''

А если в /home/astenix/newFolder/File.fn уже есть реальный файл — stow откажется создавать симлинк или потребует разрулить проблему.

источник истины —это каталог mySettings в репозитории (он же сделанный нами каталог stow). Если мы случайно грохнем каталог-источник истины, то все симлинки из него в системе превратятся в тыкву.

===== В файле ~/.bashrc =====

Какие-то настройки вынесены в отдельный файл ''bashrc.local'' 
Их можно импортировать в ~/.bashrc:

''# общий код''
''source $HOME/mySettings/bash/bashrc.local''

Если файла не будет — произойдет ошибка и чтение файла будет прервано.

Или так

''# локальные отличия''
''[ -f $HOME/mySettings/bash/bashrc.local ] && source $HOME/mySettings/bash/bashrc.local''

Тут сперва происходит проверка существования файла, и если его нет, процесс не упадёт.
